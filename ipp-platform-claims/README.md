# ipp-platform-claims

**Purpose**: Tenant resource claims — the GitOps-controlled store of all infrastructure requests made by business units against the IPP platform API. This is the **only** repository where `@ipp-automation-bot` is the authorized merge agent. Humans open PRs; the bot validates via Conftest and merges on policy pass.

**Owner**: `@ipp-automation-bot` (merge authority) · `@ipp-platform-ops` (exception approvals)

**Argo CD Watch Scope**: Watched by `claims-appset.yaml` in `ipp-platform-control`. Every merged file in this repo is reconciled by Crossplane on the target cluster.

## Claim File Naming Convention

```
claim-{resource-type}-{bu}-{env}-{sequence}.yaml
# Example:
claim-threetierapp-demo-dev-001.yaml
```

---

## Required Annotations on Every Claim

```yaml
metadata:
  annotations:
    ipp.dhl.com/request-id: ""        # Auto-generated by IPP portal
    ipp.dhl.com/business-unit: ""     # e.g. "logistics-eu"
    ipp.dhl.com/cost-center: ""       # e.g. "CC-1234"
    ipp.dhl.com/environment: ""       # dev | staging | prod
    ipp.dhl.com/requested-by: ""      # Email of requesting engineer
```

Claims missing any of these annotations will be rejected by the Conftest pre-merge policy in `ipp-platform-policies/conftest/`.

---

## Workflow

1. Developer submits infrastructure request via IPP Backstage portal
2. Portal generates a claim YAML and opens a PR to this repo
3. Conftest validates annotations, resource limits, and namespace targets
4. `@ipp-automation-bot` merges on policy pass; Argo CD picks up the change
5. Crossplane reconciles the claim into real infrastructure

---

## Branch Protection

- Branch: `main`
- **Only `@ipp-automation-bot` may merge to `main`**
- Human PRs require review from `@ipp-platform-ops` before bot merge
- No direct pushes; no force-push
